# Стабильный запуск next dev: команды и чек-лист

**Сейчас порт 3000 занят:** процесс **node**, **PID 4899**. Останови его:
```bash
kill 4899
```
Проверь, что порт свободен: `lsof -i :3000` (пусто). Затем: `rm -f .next/dev/lock` и `npm run dev`. Смотри порт в строке **Local:** в логе.

---

## 1) Кто занимает порт 3000

В терминале выполни:

```bash
lsof -i :3000
```

- **Вывод пустой** — порт 3000 свободен.
- **Есть строки** — в колонке **PID** число (например `15165`), в колонке **COMMAND** — процесс (`node` или `next`). Это и есть процесс, слушающий 3000.

Пример вывода:
```
COMMAND   PID USER   FD   TYPE DEVICE SIZE/OFF NODE NAME
node    15165  user   21u  IPv4  ...      0t0  TCP 127.0.0.1:3000 (LISTEN)
```
→ PID **15165**, процесс **node** (это и есть `next dev`).

---

## 2) Безопасно ли убивать

Да. Если на 3000 висит **node** или **next** — это dev-сервер этого проекта. Его можно завершить: данные не теряются, база (prisma/dev.db) не затрагивается.

Не убивай процессы, если в COMMAND указано что-то другое (не node/next), пока не разберёшься.

---

## 3) Точная команда kill

Подставь **реальный PID** из вывода `lsof -i :3000` (одно число во второй колонке).

Убить один процесс по PID (подставь свой PID вместо `15165`):

```bash
kill 15165
```

Если не помогло (процесс остался), принудительно:

```bash
kill -9 15165
```

Убить **все** node-процессы, слушающие 3000 (macOS):

```bash
kill $(lsof -t -i :3000)
```

Принудительно, если не реагирует:

```bash
kill -9 $(lsof -t -i :3000)
```

---

## 4) Проверить, что порт 3000 свободен

```bash
lsof -i :3000
```

Должно быть пусто (ни одной строки). Тогда порт свободен.

Дополнительно — запрос на 3000 не должен открываться:

```bash
curl -sS -o /dev/null -w "%{http_code}" --connect-timeout 2 http://127.0.0.1:3000/ || echo "connection failed"
```

Код `000` или «connection failed» — порт свободен/ничего не слушает.

---

## 5) Запуск next dev и порт

В каталоге проекта:

```bash
cd /Users/admin/Desktop/cursor/twinworks
npm run dev
```

В логе смотри строки после «Local:»:

- **Local: http://127.0.0.1:3000** — dev стартовал на **3000**.
- **Port 3000 is in use ... using available port 3001 instead** и **Local: http://127.0.0.1:3001** — стартовал на **3001**.

Порт = тот, что указан в **Local:** в этом запуске. Все проверки (браузер, curl) делай по этому порту.

---

## 6) Чек-лист

### При 404 в dev всегда проверять

1. **Порт** — открываешь и дергаешь тот же порт, что в логе dev (`Local: http://127.0.0.1:XXXX`). Не 3000, если в логе написано 3001.
2. **Один dev** — не запущено два `npm run dev`. В логе не должно быть: `Unable to acquire lock at .../.next/dev/lock`. Если есть — убить старый процесс (п.3), потом снова `npm run dev`.
3. **Кэш** — при странных 404: остановить dev (Ctrl+C), выполнить `rm -rf .next`, затем снова `npm run dev`.

### Как не перепутать порт

1. После `npm run dev` не закрывать вывод — в нём указан порт (Local: ...).
2. Проверять маршруты так: `curl http://127.0.0.1:ПОРТ_ИЗ_ЛОГА/api/probe` (подставить порт из лога).
3. В браузере открывать тот же URL с тем же портом, что в логе.

### При EMFILE (too many open files)

1. **Увеличить лимит** в этой сессии терминала (macOS):
   ```bash
   ulimit -n 10240
   ```
   Потом в том же терминале: `npm run dev`.
2. **Закрыть лишние приложения** (IDE, браузеры с десятками вкладок, другие dev-серверы), затем снова запустить dev.
3. **Перезапуск** — остановить dev (Ctrl+C), подождать 2–3 сек, снова `npm run dev`. После перезапуска watcher создаёт меньше дескрипторов, EMFILE может пропасть.

---

Итог: перед каждым «нормальным» запуском — освободить 3000 (или убить старый dev по PID), проверить порт через `lsof -i :3000`, затем запустить `npm run dev` и смотреть порт в логе. Проверки делать только по этому порту.
